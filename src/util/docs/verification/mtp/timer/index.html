1.Introduction

1.1) Purpose and Scope

This document provides a test plan for Timer Module .Timer Module provides following functions:

  • startTimer
  • StopTimer

1.2) References

1.3) Definitions, Acronyms and Abbreviations

2.Test Items

  • Time to start a Timer
  • Time to stop a Timer
  • Total timeOut time for each timer
  • Maximum number of Timers which can be started.
  • Memory Usage.
  • Timer Precision time

3. Features to be tested

1.Functional Testing:

StartTimer()

  • to test Inputs .Check for which are mandatory fields,which can be null ,which cant be null .
  • to test return value.

Stop Timer()

  • to test for Inputs . Check for which timers expired .
  • to test for return value.Whether deletion or stopping of Timer was successfull or not.

Add/Delete Functions testing:

  • adding a single node.
  • adding a node at start of linked list.
  • adding a node at end of list.
  • adding a node in the middle of list.
  • timeout of a node.
  • deletion of a node from end.
  • deletion of node from middle.
  • cyclic timers.
2.Performance testing:
  • Check for Number of threads that can be started.
  • Check for Number of timers that can be started.
  • Check for different denominations for which timer can be started.
  • Check for Time Precision in case of Timer Start time ,Timer Stop Time and TimeOut Time.
  • Check for whether each part of code is reachable or not.
  • Check Multithreading with Signals using mutex lock and unlock.
  • Check for Hash size in case of Timing Wheel implementation

4.Features Not To Be Tested

5. Approach

In this section the approach used for testing is explained.

5.1 Description

Black box testing approach is used which just checks what a program is supposed to do without going into the program structure.

5.2 Constraints

None

6. Testing Environment

The test setup is as follows:

6.1 Test SetUp Description

The Test setup consists of following:
  • Client Program:

This program is used to create a timer Handler for Timer Class. Used to create different number of threads. Used to initialise System Clock_Timer settings as per the OS used. Then start different number of timers. Check for User Timeout handler ,CallBack function. Sending messages in Queue, Receiving messages from queue. Logging time for Starttimer ,Stop timer ,TimeoutHandler to check time Precision.

  • Timer .cpp and Timer.h
  • cdsuinthash.h
  • keyAllocator.cpp and keyAllocator.h
  • TimingWheel.cpp and TimingWheel.h

6.2) Test setup variations:

There are various test scenarious:
  • We can run timer class on Different OS.

6.3 Test Menu.

The test menu provided to the user is as follows (when the "Test Client" is executed.)

1.) Create Application
2.) Create Threads
3.) Create CLOCK_TIMER time setup and environment related to OS.
4.) Start Threads
5.) Create different Timers using different threads.
6.) Create UserTimeOutFunction
7.) Create CallBackFunction
8.) Stop Timers for diffrerent threads
9.) Create different queue to recieve messages from UserTimeOutFunction.
10.) Create Signal Handler function to give Tick Count and Timeout.
11.) Exit from each thread using SIGINT function.
12.) Doing Calculations for Reports to be generated for testplans.
13.) Printing reports in a seperate file.
14.) Check above times for different Hash sizes to verify that change in hash size does not affect the timer effieiency.
15.) Verify Timer Node additions and deletions.

STARTTIMER:

Item Information
Test Case TIMER/START_TIMER/VAL_TIMECOUNTS
Purpose To test valid Timecounts value
Pre-Test Condition It is to be of type Uint16
Expected Outcome Valid Timer TimeCounts
Test Description Pass any valid Uint16 value as timecount.It works Successfully.



Item Information
Test Case TIMER/START_TIMER/VAL_TIMEUNITS
Purpose To test valid TimeUnits
Pre-Test Condition It is to be of predefined Enum CdSuTimeUnitsType only.
Expected Outcome Multiplied with TimeCounts to give total Time in MS.
Test Description Choose any type from Enum nad assign it to TimeUnits.


Item Information
Test Case TIMER/START_TIMER/VAL_TIMEOUT_REPORTTYPE
Purpose To test validity of User Handler functions or Callback functions.
Pre-Test Condition It is to be if predefined Enum type CdSuTimeoutReport Type
Expected Outcome If it is of type Message then UserTimeOut?Handler function should be there to handle the Timeout.If Its of type CallBack then CallBack function should be passed as a parameter in UserInfo while Start_Timer function is being called Else Error will come.
Test Description If Message type then use a UserTimeOutHanlder function else in case of Callback Specify the callbackfunction(CallBcakfunction should not be Null in case of CallBackFunction type.)


Item Information
Test Case TIMER/START_TIMER/VAL_OPCODE
Purpose It is passed to UserTimeOutHandler function as remote information
Pre-Test Condition It is of type Uint32 and should be provided.
Expected Outcome It is used for remote destination only and its use depends on Client .
Test Description Assign some value of type Uint32 tp opcode.


Item Information
Test Case TIMER/START_TIMER/VAL_CALLBACK_FUNCTION
Purpose To test validity of Callback functions.
Pre-Test Condition It is to be set to NULL or not defined.
Expected Outcome If it is of type Message then UserTimeOut?Handler function should be there to handle the Timeout.If Its of type CallBack then CallBack function should be passed as a parameter in UserInfo while Start_Timer function is being called Else Error will come.If CallBack function is not there then segmentation fault occurs.
Test Description If Callback do Specify the callbackfunction(CallBackFunction should not be Null in case of CallBackFunction type.)

Item Information
Test Case TIMER/START_TIMER/INVAL_TIMECOUNT
Purpose To check timeCount is invalid in what cases.
Pre-Test Condition Assign TimeCount some real or any other value other thanUint32
Expected Outcome It will give cast error and warning that assignment from double to Uint16 is not possible.It is supposed to be time and time cant be real or some negative value.Finally in NULLcase segmentation fault will occur.Returns CDSU_TMR_RV_INV_TIME_COUNT in case of 0 timecount.
Test Description assign some value real or negative value to timecount or set it to NULL

Item Information
Test Case TIMER/START_TIMER/INVAL_TIMEUNITS
Purpose To Test invalidity of timeUnits in different aspects.
Pre-Test Condition It can be NULL or some negative value or some other value.
Expected Outcome If its NULL then segmentation fault ,If its other value then error: cant convert from int to CdSuTimeUnits.Returns CDSU_TMR_RV_INV_TIME_COUNT in case of invalid timeunits
Test Description Assign some value to it.


Item Information
Test Case TIMER/START_TIMER/INVAL_TIMEOUT_REPORTTYPE
Purpose To test invalidity of User Handler functions or Callback functions.
Pre-Test Condition It is to be NULL or no defined type.
Expected Outcome If TimeoutReport type is not defined then when timeout occurs then neither UserTimeoutHandler nor CallBack function is called up.
Test Description Do not specify the TimeoutReport type and then run module.


Item Information
Test Case TIMER/START_TIMER/INVAL_OPCODE
Purpose To test use of opcode.
Pre-Test Condition It is not defined or its defined of some wrong type.
Expected Outcome If it is not defined then there is no error.But it is required at destination when userTimeOutHandler function is called.If its of some wrong type then warning that of incompatible types come.
Test Description Specify some wrong value or NULL to opcode.



Item Information
Test Case TIMER/START_TIMER/INVAL_CALLBACK_FUNCTION
Purpose To test validity of Callback functions.
Pre-Test Condition It is to be if NULL initially.
Expected Outcome If Its of type CallBack then CallBack function should be passed as a parameter in UserInfo while Start_Timer function is being called When TimeOut occurs it will search for CalBack functruion and in case it doesnt finds then segmentation fault.Returns Invalid callback .Returns CDSU_TMR_RV_INV_CALLBACK.
Test Description If Message type then use a UserTimeOutHanlder function else in case of Callback Specify the callbackfunction(CallBackfunction should not be Null in case of CallBackFunction type.)

DIFFERENT TIMER RESOLUTIONS:

Any combination of timers resolution can be choosen for starting timers.Right from 10 MS to Hour.
Resolution means timerUnits Type.

struct UserInfo has following fields:

1. Mandatory fields:

  • timeCounts : cant be 0 or NULL
  • timeUnits : cant be 0 or NULL (USER to use predefined types only)
  • timerType : in case of type callback then CallBack function has to be provided.else it will give segmentation fault.
2. Optional fields:
  • Opcode : can be null
  • desttype : depends on client.
